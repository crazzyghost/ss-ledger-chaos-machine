# ADR-014 — Field-descriptor flow catalog + client-side inference for the Single Flow Run form

- **Status:** Accepted
- **Date:** 2026-06-23
- **Phase:** [011 — Single Flow Run Redesign](../phases/011-single-flow-run-redesign/DESIGN.md)
- **Supersedes:** the flat-string-list shape of `FlowCatalogEntry` introduced in
  [Phase 003 / task 002](../phases/003-transaction-flow-engine/002-single-transaction-publishing-api.md)
  (extends it; the legacy lists remain derivable — see Decision 1).

## Context

The idea `004_single_flow_run.md` reworks the chaos **Single Flow** runner into a
**Single Flow Run** console: a radio-selected transaction type, a two-column layout
(transaction form left, chaos options right), and — critically — a form that distinguishes
**required-but-prefilled** fields (shown) from **non-required-but-inferable** fields
(collapsed by default), with concrete prefill/inference rules:

- Transaction request id → **autogenerated UUID**.
- `source_va_id` / `destination_va_id` → pickers scoped to an **account kind**
  (Organization vs System) — system slots prefilled from the chart of accounts.
- `amount` → default `1000.0000`.
- `organization_id` (or source/destination org id) → **inferred from the selected VA**.
- `currency` → **inferred from the source VA**.
- `tenant_id` → **inferred from the source VA when its ownership is ORGANIZATION**.

The current catalog DTO (`FlowCatalogEntry`) exposes only `requiredFields: List<String>`,
`optionalFields: List<String>`, `csvColumns`, and `partitionKeyField` — bare field-name
strings with **no per-field metadata** (type, default, collapsed state, account kind, or
inference rule). The frontend renders every field as an undifferentiated text input. There
is nowhere to express "this field is a UUID we autogen", "this is a System-account VA
picker", or "this is collapsed and inferred from the source VA". Two questions must be
settled before the redesign:

1. **Where does the per-field UI metadata live** — server (catalog) or client (hard-coded)?
2. **Where does the inference run** — server (a new lookup endpoint) or client (off the
   already-loaded VA object)?

## Decision

### 1. Enrich the catalog with structured per-field descriptors (additive, backward-compatible)

Extend `FlowCatalogEntry` with a `List<FlowFieldDescriptor> fields`, where each descriptor
carries the UI/semantic metadata the form needs:

| Attribute | Meaning |
|---|---|
| `name` | wire/payload field name (snake_case), e.g. `topup_request_id`, `completed_by` |
| `label` | human display label, e.g. "Transaction Request ID", "Initiated By" |
| `kind` | `TEXT` \| `UUID` \| `AMOUNT` \| `DATETIME` \| `SELECT` \| `VA_REF` |
| `required` | shown by default and validated before publish |
| `advanced` | non-required **collapsed-by-default** ("needs input but can be inferred") |
| `defaultValue` | seed value (e.g. amount `1000.0000`; treasury channel defaults) — nullable |
| `autogen` | `NONE` \| `UUID_V4` — client seeds and can regenerate |
| `inference` | `NONE` \| `ORG_FROM_SOURCE_VA` \| `ORG_FROM_DEST_VA` \| `CURRENCY_FROM_SOURCE_VA` \| `TENANT_FROM_SOURCE_VA` |
| `accountKind` | for `VA_REF` only: `ORGANIZATION` \| `SYSTEM` (drives picker filtering) — nullable |
| `slotName` | for `VA_REF` only: the flow slot this field fills (system VAs prefill from CoA) — nullable |
| `options` | for `SELECT` only: allowed values (e.g. treasury channel `bank`/`momo`) — nullable |

`requiredFields` / `optionalFields` / `csvColumns` / `partitionKeyField` **remain on the
record** and are derived from `fields` (`required==true` → required list; the rest →
optional; `csvColumns` keeps its current ordering). This keeps the CSV batch upload page,
existing catalog tests, and any other consumer working unchanged — the redesign is purely
additive on the wire.

The catalog stays the **single source of truth** for form shape: the frontend renders
whatever the descriptors say, so adding the deferred flows (Settlement/Collection/
Disbursement) later is a server-side descriptor change with no UI rework.

### 2. Mark which flows the Single Flow Run exposes via a `runnerVisible` flag

Add `boolean runnerVisible` to `FlowCatalogEntry`. The five idea-listed types
(`TOPUP_CONFIRMED`, `TRANSFER_REQUESTED`, `TREASURY_SWEEP_COMPLETED`,
`TREASURY_PREFUND_COMPLETED`, `TREASURY_TRANSFER_COMPLETED`) are `true`; all others
(including `ORGANIZATION_ONBOARDED` and `ORGANIZATION_VA_UPDATED`, which the idea removes,
and the deferred Settlement/Collection/Disbursement flows) are `false`. The runner renders a
radio of `runnerVisible` flows. Re-introducing a deferred flow later (per the user's
"5 now, others later" decision) is a one-line flag flip plus its descriptors — **no UI
change**.

### 2a. VA ids route through `slotOverrides` of **seeded** slots; org-account slots must be added

Tracing the flow builders (`TopUpConfirmedFlowBuilder`, `TransferRequestedFlowBuilder`,
`Treasury*FlowBuilder`) showed `source_va_id`/`destination_va_id` are read **only** from
`ctx.resolvedSlots().getOrDefault("source"/"destination", "")` — never from `flowFields`. And
`SlotResolver.resolveAll` iterates **only slots that have a `flow_slot_config` row**, so a
`slotOverrides` entry for an unconfigured slot is silently discarded. The current seed
(`chaos-bootstrap.yml`) configures only **system**-account slots and is **missing** the
user-supplied Organization-account slots (`TOPUP_CONFIRMED.source`,
`TRANSFER_REQUESTED.source`+`.destination`) and the `TREASURY_TRANSFER_COMPLETED` slots
entirely. **Decision:** a `VA_REF` descriptor declares a `slotName`, the form always emits the
picked VA as `slotOverrides[slotName]`, and Phase 011 **seeds the missing slot rows**
(Organization slots with `accountRole: null` — request-only; Treasury-Transfer with system
roles). Without this, the redesign's pickers publish empty VA ids. This is a config/seed
change, not a schema or endpoint change.

### 2b. Collapsed-but-builder-required fields must carry a default or inference

Builders call `FlowFields.getRequired(...)` (HTTP `400` on blank) for fields the idea marks
non-required/collapsed (`organization_id`, `approved_by`, `narrative`, `initiated_by`,
`completed_by`, treasury channels, `currency`). **Decision:** every such field's descriptor
must guarantee a value even when collapsed — via `autogen`, `inference`, or a `defaultValue`
seeded from the bin-script samples (e.g. `approved_by=ops@acme.example`). "Non-required in the
UI" means "auto-filled and collapsed", **not** "may be absent from the payload".

### 3. Inference runs client-side off the selected VA object — no new backend endpoint

`VirtualAccountResponse` already carries `ownershipType` (ORGANIZATION/SYSTEM),
`organizationId`, and `currency`. The Single Flow Run already loads VAs to populate the
`VA_REF` pickers. Therefore each `inference` rule is computed **in the browser** from the
selected VA object the moment it is picked:

- `ORG_FROM_SOURCE_VA` / `ORG_FROM_DEST_VA` → `selectedVa.organizationId`
- `CURRENCY_FROM_SOURCE_VA` → `sourceVa.currency`
- `TENANT_FROM_SOURCE_VA` → `sourceVa.organizationId` **iff** `sourceVa.ownershipType === "ORGANIZATION"` (else left blank)

No server round-trip, no new lookup API. The catalog only *declares* the rule; the client
*applies* it. Inferred values populate the collapsed "advanced" inputs and stay
**user-overridable** (the operator can expand and edit them — this is a chaos harness, so
sending a wrong/empty inferred value on purpose must remain possible).

### 4. Transaction request ids are autogenerated UUID v4 client-side

The request-id field of each flow (`topup_request_id`, `transfer_request_id`,
`{sweep|prefund|transfer}_request_id`) is `kind=UUID`, `autogen=UUID_V4`: the client seeds a
fresh v4 on type-select and offers a **regenerate** affordance. This realizes the
"autogen UUID" requirement of idea `004` and the spirit of idea `003_auto_generated_uuid`.
**Deferred:** idea `003`'s "transaction ids should be selectable" (re-using a prior run's id
from history to exercise idempotency) is out of scope here and noted as a follow-up.

## Consequences

**Positive**

- One declarative contract drives the whole form; server owns flow shape, client owns
  rendering. Adding the deferred flows is server-only.
- Zero new backend inference surface; inference is a pure function of an object the client
  already holds. Fast, no extra latency, no new failure mode.
- Backward compatible — legacy list fields persist, so CSV/batch and existing tests are
  untouched.

**Negative / trade-offs**

- Inference logic lives in the client, so it is **not** enforced server-side: a hand-crafted
  `POST /flows/{type}` can still send an org/currency that disagrees with the VA. Acceptable
  — and arguably desirable — for a chaos harness; the server validates only structural
  required fields (unchanged).
- The descriptor model couples catalog evolution to a richer schema; adding a new `kind` or
  `inference` rule touches both the Java enum and the TS renderer. Mitigated by keeping both
  small and centrally mapped.

**Open items (flagged, not blocking)**

- **Label vs wire-name mismatches** between the idea and existing payloads — Inter-VA's
  "Source Payment Reference" vs payload `narrative`; Treasury's "Initiated By/At" vs payload
  `completed_by`/`completed_at`. The `name` (wire) vs `label` (display) split absorbs this:
  we keep the wire field names and map the idea's labels onto them. Confirm the labels with
  the product owner; do not rename wire fields (they are the ledger contract).
- **`idempotency_key` field has no transport (decision needed).** The idea lists "Idempotency
  Key" as a collapsed field, but `PublishFlowRequest` has no such field and every builder
  hard-derives `metadata.idempotency_key` = `"<event-type>:" + eventId`. Options: **(i)** drop
  it from the form (server-derived; show read-only/derived), or **(ii)** add an optional
  `idempotencyKey` to `PublishFlowRequest` and have builders honour it when present —
  **recommended**, because forcing a specific key across *separate* runs is a real
  idempotency-chaos capability the `duplicate` strategy (same-run only) cannot exercise. Until
  decided, the descriptor for `idempotency_key` is omitted from the catalog.
- **Treasury org-id fields are inert.** The idea lists "Source/Destination Organization ID"
  for treasury, but `Treasury*CompletedEventData` has no such fields and the builders never
  read them — those descriptors are omitted (org-id inference applies to Top-up + Inter-VA
  only).
